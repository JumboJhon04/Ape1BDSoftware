Uniones Relacionales (Joins) y Patrones

Dominar los Joins va más allá de unificar tablas. Implica entender cómo el motor de base de datos construye productos cartesianos y los filtra basándose en condiciones matemáticas.

Patrones de Uniones

  • INNER JOIN: Intersección. Omite todo lo que no hace match.
  • LEFT JOIN: Base + coincidencias. Es la piedra angular para construir el Anti-Join.
  • SELF JOIN: Unir una tabla consigo misma (Ej: FROM empleados e1 JOIN empleados e2 ON e1.jefe_id = e2.id). Esencial para resolver jerarquías en una sola tabla.

El Producto Cartesiano

Olvidar la cláusula ON (o usar CROSS JOIN) genera la multiplicación de las filas. Una tabla de 1,000 filas cruzada con otra de 1,000 generará 1,000,000 de registros en memoria RAM al instante.

El Patrón "Anti-Join"

¿Cómo encuentras clientes que NUNCA han comprado? El uso de NOT IN es ineficiente y puede fallar con valores NULL. El Anti-Join es el estándar de alto rendimiento:

SELECT c.nombre
FROM clientes c
LEFT JOIN facturas f ON c.id = f.cliente_id
WHERE f.id IS NULL;

* Trae todos los clientes, intenta cruzarlos, y luego filtra quedándose solo con los que fallaron en cruzar (las columnas de la factura estarán en NULL).

Eje 2

Arquitectura de Filtros y Rendimiento

Evaluación: EXISTS vs IN

Para validar si un dato existe en otra tabla inmensa:

  • Operador IN: Construye toda la lista de resultados en memoria antes de comparar. Es ineficiente para tablas grandes.
  • Operador EXISTS: Utiliza evaluación de cortocircuito (Short-circuit). Si encuentra una sola coincidencia, devuelve TRUE y aborta la búsqueda instantáneamente.

Orden de Ejecución Lógica

1. FROM / JOIN
2. WHERE (Filtro nivel registro)
3. GROUP BY
4. HAVING (Filtro nivel grupo)
5. SELECT
6. ORDER BY / LIMIT

SARGability y Destrucción de Índices

Una consulta es SARGable (Search Argument Able) si el motor puede usar un índice B-Tree para encontrar los datos rápido.

❌ Anti-Patrón (Table Scan)

Aplicar funciones a la columna rompe la SARGability. El motor debe calcular la función fila por fila.

WHERE YEAR(fecha) = 2026
WHERE UPPER(nombre) = 'JUAN'

✅ Patrón Óptimo (Index Seek)

Aísla la columna en el lado izquierdo. Haz las conversiones matemáticas en el valor estático de la derecha.

WHERE fecha >= '2026-01-01' AND fecha < '2027-01-01'
WHERE nombre = 'JUAN' (con Collation CI)
Eje 3

Agregaciones y Pivotes Matriciales

Las consultas de "análisis de resultados" exigen extraer inteligencia de negocio. Aquí la matemática de conjuntos y el comportamiento de NULLs son críticos.

El Peligro de los Valores NULL

Si tienes valores [10, NULL, 20]:

  • SUM() = 30 (Ignora NULL).
  • COUNT(*) = 3 (Cuenta todas las filas físicas).
  • COUNT(col) = 2 (Cuenta solo las filas no nulas).
  • AVG() = 15. Porque hace 30/2. ¡Atención! Si el NULL era un "cero matemático", el promedio real debería ser 10 (30/3). Usa COALESCE(col, 0) para protegerte.

Pivoteo Manual de Datos

A menudo los reportes exigen convertir "filas" de categorías en "columnas". El estándar SQL ANSI para lograr esto sin la cláusula propietaria PIVOT es combinando Agregaciones con expresiones condicionales CASE.

reporte_pivoteado.sql
-- Obteniendo total de facturas por estado en 1 sola fila
SELECT
  id_vendedor,
  SUM(CASE WHEN estado = 'PAGADA' THEN 1 ELSE 0 END) AS pagadas,
  SUM(CASE WHEN estado = 'ANULADA' THEN 1 ELSE 0 END) AS anuladas
FROM facturas
GROUP BY id_vendedor;
Eje 1.5

Joins: Optimizador y Reconciliación

La Tabla Conductora (Driving Table)

Cuando haces múltiples joins (ej: A JOIN B JOIN C), el motor no procesa todo a la vez. El Query Optimizer escoge una tabla base para arrancar llamada Driving Table.

  • El motor intenta predecir qué tabla tiene menos registros mediante el uso de "estadísticas".
  • Regla de oro de Arquitectura: Filtra agresivamente (con WHERE) tu tabla principal ANTES de hacer joins masivos. Mientras menos filas inicies en la conducción, más rápido evaluará los cruces subsecuentes.

El Poder del FULL OUTER JOIN

Mantiene TODOS los registros de la izquierda y TODOS los de la derecha. Si no hay match en algún lado, pone valores NULL.

Caso Analítico Top: Reconciliación de datos. Es la forma definitiva de comparar inventarios (Ej: Lo que está en Sistema vs Lo que está en Bodega) para detectar qué falta y qué sobra en una sola consulta.

SELECT sys.id, fis.id
FROM inv_sistema sys
FULL OUTER JOIN inv_fisico fis
ON sys.id = fis.id
WHERE sys.id IS NULL OR fis.id IS NULL;
Eje 3.5

Técnicas Híbridas y Textuales

El Costo de COUNT(DISTINCT)

Contar valores únicos (ej. Usuarios distintos activos al mes) exige que el motor ordene en memoria RAM toda la columna para poder eliminar los duplicados.

En entornos de Big Data, usar COUNT(DISTINCT id) provocará un desbordamiento de memoria masivo (Spill to TempDB). En analítica avanzada, a menudo es mucho más rápido hacer una subconsulta con un GROUP BY id inicial y luego ejecutar un COUNT(*) externo sobre ese set ya reducido.

Agregación de Cadenas (Strings)

¿Qué pasa si en lugar de sumar dólares, necesitas colapsar texto? Por ejemplo: Mostrar un Cliente y en una sola celda listar todos los nombres de los productos que compró separados por comas.

-- PostgreSQL / SQL Server Moderno
STRING_AGG(producto, ', ')

-- MySQL / MariaDB
GROUP_CONCAT(producto SEPARATOR ', ')

Transforma las filas múltiples (Manzana, Pera, Uva) en un valor unificado: "Manzana, Pera, Uva". Una de las funciones más requeridas en exportaciones de reportes gerenciales.
